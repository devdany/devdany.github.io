이더리움 네트워크 노드를 셋팅하고 사용해보자!

  • dany
  • 2017/12/15
  • BlockChain, Ethereum
Categories: BlockChain, Ethereum

Geth

Geth는 Ethereum에서 공식 제공하는 노드라고 할 수 있습니다. 즉, Geth를 설치하면 그 PC는 이더리움 네트워크의 노드가 됩니다. Geth를 설치하고 처음 실행하면 1번 현재까지의 모든 블록정보를 Import하기 시작할 것입니다.

처음의 저처럼 블럭체인 공부를 시작하는 사람이라면 일단 설치하고 볼 수 있겠지만, 미리 경고하지만 이더리움 네트워크의 노드가 되어 모든 블럭정보를 내려받는 것은 조금 주의깊게 결정할 필요가 있습니다.

저는 현재 총 몇 기가바이트가 사용될 지 예측하지 못하고, 또 내려받는 동안 메모리 사용도 결코 만만치 않습니다. 따라서 노트북과 같이 발열관리가 좋지 못하거나, 용량이 충분치 않은 PC에는 geth를 설치하지 맙시다.

저도 노트북으로 현재 포스트를 쓰고 있기 때문에, testRPC라는 개발환경 geth를 설치하겠습니다. 이 testRPC는 geth와 같은 통신 API를 제공하지만, 메인네트워크가 아닌 로컬 혼자만의 이더리움 네트워크 환경이라고 생각하시면 됩니다.

트랜잭션을 발생시키면 그 즉시 블럭이 생성되고 결과값이 리턴되며, 테스트 계정 열개와 각각의 계정에 100eth씩 주기 때문에 개발환경으로 아주 적합하다고 개인적으로 생각합니다.

main net 환경에서 이더리움 네트워크를 사용하고 싶으신 분은,

https://github.com/ethereum/go-ethereum/wiki/Building-Ethereum

이 곳을 참고하셔서 설치하시면 됩니다.

그럼 전 testRPC설치를 해보겠습니다. (맥환경 기준이니 참고하시기 바랍니다. 맥에서는 xcode가 설치되어 있어야 합니다. 윈도우 환경에서서는 C++ 컴파일러가 설치되어 있어야 함을 미리 알려드립니다.)

또 저는 설치환경으로 npm을 사용하기 때문에, node가 설치되어있어야 합니다.

npm install -g ethereumjs-testrpc

설치가 이상없이 잘 되었으면, 이제 커맨드창에서 testrpc를 입력해 봅시다.

이 화면이 보인다면 설치가 잘 된것입니다. 보시는 대로 10개의 계좌가 들어있고, 각 계좌에는 100eth가 들어있습니다. 원래 서버 프로그래밍을 하시던 분이라면, 테스트용 로컬서버가 설치되었다고 생각하시면 편하실 것같습니다.

그렇다면, 이제 이 서버와 통신하는 방법을 알아 볼 차례입니다. 방법은 여러가지가 있지만, 가장 좋은 방법은 json-rpc 방식의 통신 api를 활용하는 것입니다.

이제부터 편의상, 이 testRPC를 geth라고 칭하겠습니다. 사실 통신방법이가 기능은 동일하기 때문에 이더리움 메인넷에서의 트랜잭션을 확인할 수 없다는 것 외에는 동일하게 사용할 수 있기 때문입니다.

geth는 json-rpc 방식의 통신 api를 지원하고 있으며, 이더리움 node와의 통신 api도 (대표적으로 web3, 후에 다뤄보겠습니다.) 언어만 다를뿐 내부적으로는 이 방식을 취하고 있습니다.

그래서 먼저 json-rpc 방식으로 방금 설치한 이 테스트용 geth노드와 통신해 보려고 합니다.

당연한 이야기지만, json을 모른다면 먼저 json을 아셔야 합니다. 일단 다음 방법으로 패스하시기 바랍니다.

이더리움에서는 account를 생성하면, 그 account는 물리적인 그 노드(geth)에 저장됩니다. 물론 생성시 발급되는 privateKey만 알고 있으면 다른 node에서 account를 import할 수 있습니다.

일단 이 노드에 저장된 account와(정확히는 testRPC구동시 최초로 생성된 account), 그중 한 account에 저장된 balance를 가져와 보겠습니다.

먼저 ethereum geth json-rpc api문서를 봅시다.

https://github.com/ethereum/wiki/wiki/JSON-RPC 앞으로 개발을 할때, 시도때도 없이 들어가게 될 곳입니다.

이렇게 어떤 양식으로 json-rpc통식을 보내줘야 하는지, 그러면 어떤 리턴을 주는지가 정의되어 있습니다.

그러면, 보내봅시다. 저는 자바로 클라이언트 프로그램을 간단하게 짜서 돌려보았습니다.

그다지 복잡하지는 않은 코드이므로 간단하게만 설명드리자면, 이 geth의 default 포트인 8545로 커넥션을 객체를 얻고, 자바에서 객체를 json형태로 변환할 때 많이 쓰이는 라이브러리인 Gson을 사용하여 위 양식에 맞게 json을 생성하여 커넥션을 통해 geth로 request를 보내고, response를 받는 코드입니다. (request / response라는 용어 사용이 부적절하다고 포스팅한적이 있습니다만, 그것은 많은 노드들로 구성된 이더리움 네트워크에 대해서는 부적절하지만, 하나의 노드에 대한 api 콜에 대해서는 적절하다고 생각합니다.)

실행 결과로 이런 결과를 얻을 수 있었습니다.

위 testRPC실행시 확인한 account주소들과 확인해보면, 일치함을 알 수 있을것입니다.

geth 사용에 대해서는 일단 이 정도로 정리하도록 하겠습니다.

MetaMask

아마 이더리움 공부를 시작하실때, 예제로 가장 많이 사용되었던 걸로 기억하는 메타마스크 입니다. 이것은 크롬 브라우저 플러그인 애플리케이션으로서, 이더리움 지갑입니다.

이 메타마스크를 사용하시면, geth설치 없이도 이더리움 네트워크상에서의 왠만한 모든 일들을 할 수 있습니다.

그런데 이상합니다. 설치하시고 사용해보시면 아시겠지만, 가볍기 그지없는 이 어플리케이션이 어떻게 그 어마어마한 노드없이 이더리움 노드로서의 기능을 다 할 수 있는걸까요?

이미 예상한 분도 있으시겠지만, 이 메타마스크 역시 위에서 제가 짠 자바 클라이언트 프로그램과 원리는 크게 다르지 않습니다. 즉, 다른 곳에 Provider를 셋팅하고 그 쪽을 바라보도록 해둔 프로그램인 것입니다.

실제로, 이것저것 만지다 보면, 메타마스크에서 따로 셋팅한 Provider를 바라보도록 설정할 수 있음을 금방 아실수 있을겁니다.

즉, 내가 설치한 메타마스크를 통해 생성한 account는 사실 메타마스크가 바라보는 Provider에 들어 있는겁니다.

메타마스크 설치와 사용에 대해서는 다른 포스팅도 많고, 또 사용법이 무척 간단하므로 따로 쓰지는 않겠습니다.

Infura

아마 서버개발자분이라면 aws는 한번쯤 다 들어보셨을 겁니다. infura는 이더리움 provider의 aws라고 보시면 바로 이해하실 거라고 믿습니다.

즉, 개인용 노드 무료 호스팅 업체입니다. 하지만 이곳에 account를 생성하는 등의, 보안상 문제가 될 수 있는 api는 사용할 수 없도록 해두고 있어 개인적으로는 선호하지 않습니다.

하지만 단순히 이더리움 full노드로 부터 데이터를 가져오기만 하는 용도로는 간편하기도 하고 비용도 들지 않아 좋습니다.

사용해보고 싶으신 분은, 구글에 infura라고 검색하셔서 회원가입하신 후 사용하시면 됩니다.

다음에는, 좀더 간편한 node와의 통신방법, 그리고 이더리움에서 사용하는 단위에 대해서 포스팅해보겠습니다.
